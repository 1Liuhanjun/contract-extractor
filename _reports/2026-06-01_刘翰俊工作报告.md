# 合同字段提取系统 — 开发工作报告

**日期**：2026年6月1日  

---

## 一、项目概述

将物流运输合同 OCR 后的非结构化文本，通过 **两阶段 LLM 提取架构**，自动填充到固定的 Excel 台账表格中。目标替代人工逐份签读合同、逐字段手填的重复劳动。

---

## 二、核心架构

```
合同原文（OCR .docx）
    ↓
Stage 1: LLM 事实抽取 → 原始事实 {value, evidence}
    ↓
字段知识库 + 少样本实例 → 映射规则
    ↓
Stage 2: LLM 标准化映射 → 标准字段值
    ↓
Excel 台账输出
```

**关键设计决策**：分离"阅读理解"和"业务映射"两个阶段。Stage 1 只问"合同写了什么"，Stage 2 问"业务人员会填什么"。避免之前单一 prompt 把两种能力混在一起导致的问题。

### 3.4 灵活的内置内操验证（验证规则优化）

- `/` 与空字符串视为等价
- `否` 与 `/`、`"否"` 与空视为语义等价
- 百分比与小数互转（`9%` vs `0.09`）
- 数值精度差异容忍（`317.842` vs `317.8`）
- 客户名称、旺季字段支持模糊匹配（公共子串 ≥ 60%）
- 所有比较都不区分空格差异

### 3.5 LLM 复核功能

- ✅ 读取合同原文，逐字段判断差异原因
- ✅ 自动分类：AI 遗漏 / 合同不存在 / 参考答案错误 / AI 提取正确
- ✅ 每条差异附带合同原文依据

## 四、技术问题修复记录

| # | 问题 | 根因 | 状态 |
|---|---|---|---|
| 1 | `'ascii' codec can't encode` | httpx 中文编码检测失败 | ✅ 换 urllib + 强制 UTF-8 |
| 2 | DeepSeek V4 Pro 返回空内容 | 默认思考模式，内容在 reasoning_content | ✅ 禁用 thinking + fallback |
| 3 | JSON 截断 | max_tokens=4096 不够 Stage 1 | ✅ → 8192 |
| 4 | API 400 Bad Request | URL 路径错误、response_format 不支持 | ✅ 修正 URL + 去掉该参数 |
| 5 | 验证参考值全空 | Excel 表头带 `*` 号 | ✅ 自动 lstrip("*") |
| 6 | 参考答案读错 Sheet | 活动表是"邮政"不是"新合同台账" | ✅ 强制读最后一个 Sheet |
| 7 | `_handle_upload` 不清空旧文件 | 多次上传混在一起 | ✅ 上传前先清空 |
| 8 | 空结果时导出崩溃 | `_send_json` 后缺 `return` | ✅ 已补 |
| 9 | 中文文件名报 latin-1 错误 | HTTP 头只接受 ASCII | ✅ filename*=UTF-8'' 编码 |
| 10 | LLM 复核不发合同原文 | 前端传假文本 "请参考合同" | ✅ 提取结果附带原文 + 前端回传 |
| 11 | LLM 复核需重新上传文件 | 合同原文没保存 | ✅ 存入 processResults.contract_text |
