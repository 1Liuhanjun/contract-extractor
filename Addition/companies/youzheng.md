# 邮政合同字段提取 Skill

> 用途：注入 DeepSeek（纯文本）prompt 的字段抽取规则。输入 = 某份邮政合同的 OCR 文本（带页码标记）+ 价格/系数表的二维单元格结构。输出 = 结构化字段（Pydantic schema）。
> 适用：中国邮政线路运输服务合同（一干/二干/计趟/计重，多省份多年份多模板）。
> **本文是「邮政」表（线路明细表）处理全流程的单一规则文档**，运行时每次随抽取 prompt 注入。§0~§5 是给大模型的**抽取**铁律；§6 是「邮政」表 29 列的**逐列权威口径速查**；§7 是抽取之后由**代码层**执行的换算/派生/校验口径（大模型不执行，仅供核对、与代码保持一致）。
> 字段是否抽取的机器权威是 `field_map.yaml`（与本文 §6 一一对应、互为镜像，改动须同步）；模板红字表头与 `field_map.yaml` 的 `extract:false` 由 `tests/test_template_redfields.py` 守恒校验。

---

## 0. 第一原则：语义为主，位置线索为辅且永远可被推翻（最重要）

合同模板因省份、年份、业务类型差异极大。为兼顾效率与泛化，本 Skill 对部分字段给出**「常见出现位置」作为优先查找线索**——但它只决定**查找顺序**，绝不决定字段是否存在。使用方式固定为三步：

> **优先位置先看 → 该位置没有就全文检索 → 全文也没有才留空。**

因此请把所有位置线索当作"先看这里更可能命中"的提示，而不是"只能在这里找"的限制。**字段在哪、以什么措辞出现，最终由你的语义理解决定。**

铁律：

1. **位置线索不是硬过滤。** 优先位置找不到时**必须**继续全文查找，**禁止**因为"该出现的位置没有"就直接判定缺失。
2. **位置线索不是默认值。** 任何字段都不要因为"通常是某值/通常在某处"而填入；文中没有就是 null。兜底默认由系统配置决定，不归你管。
3. **只信本合同文本。** 每个值必须来自当前这份合同并附原文证据；禁止用"邮政合同一般如何""上一份合同填了什么"来推断。
4. **没有证据就留空。** 找不到、看不清、无法确定 → `value=null` + 低置信。**留空永远优于猜对。**
5. **不迁移。** 把每份合同当作第一次见到的新模板，只读它写了什么；不要把任一合同的特征带到另一份。
6. **不脑补 OCR。** 输入是 OCR 结果，可能有错字/串行/字母混入数字。可疑内容标低置信并在证据说明，**不要**改成看起来合理的值。

---

## 1. 输出契约

每个字段输出四元组：

```json
{ "value": <值或 null>, "confidence": <0~1>, "evidence": "<本合同原文片段>", "page": <页码或 null> }
```

- `value=null`：合同中没有、看不清、或无法确定。
- `confidence`：对"该值来自本合同且正确"的把握；证据清晰唯一→高，模糊/多处不一致/OCR 可疑→低（≤0.7）。
- `evidence`：**必须是 OCR 文本里真实出现的片段**（系统会回查，编造的引用按幻觉丢弃）。
- 多候选无法判定 → 全部列出 + 低置信，交复核。

---

## 2. 输出格式归一化（只规范"怎么写出来"，不规范"去哪找"）

- 百分比 → 小数：如税率"9%"写 `0.09`。原文是区间/多值（如"10%/15%"）则**保留原文字符串**。
- 日期 → `YYYY-MM-DD`。无法解析的口语化时间（如"签订之日起"）原样保留。
- 金额按目标列要求的单位；与原文单位不同则换算并在证据注明原文单位，不确定则保留原值 + 低置信。
- 区分两种"没有"：合同**明确**写了"无/不收取/不适用" → 按字段填 `/` 或对应表述；只是**没提到** → `null`。

---

## 3. 红字表头规则（硬业务约束，必须遵守）

- 字段映射 `field_map.yaml` 中标 `extract:false` 的列（项目编号、合同编码、流程状态类、快递单号、钉钉单号等）→ **一律不抽**，由系统留空。合同里即使出现疑似值也不要填。
- **邮政线路明细表里的红字列**：`类型`、`备注`、`合同编号`、`元/趟/条/40吨B` 四列也是 `extract:false`（业务规则说明书红色高亮）——**一律不抽、留空**：类型/备注由人工标注，合同编号去重改用文件哈希，40吨B 不参与。
- **例外**：邮政明细表的 `分包号` 要抽取（它是表格里的真实数据列）。
- 数据格本身字体红色 **不影响**抽取（忽略数据格颜色）。

---

## 4. 要抽取的字段（含义 + 优先查找位置）

> 完整列定义见 `需求文档.md §5.2 / §5.3`。下表「优先位置」仅是查找线索：**先看 → 没有就全文找 → 全文也没有才 null**（见 §0）。标 C（红字）由系统留空、标 D（固定值）由代码填，均不在此抽取。

**新合同台账（元信息）**

| 字段 | 含义 | 优先位置（非约束，找不到要全文兜底） |
| --- | --- | --- |
| 登记日期 | 合同签订日期 | 末页落款/签章处，其次封面 |
| 业务类型 | 一干/二干/三干 等干线类别 | 标题、首部 |
| 合同类型 | 主选/备选（依中标位次或"主/备选"表述；无依据→null，不默认） | 首部、中标相关条款 |
| 客户名称 | 甲方全称 | 首部当事人栏、末页盖章处 |
| 合同名称 | 合同完整标题 | 封面/首页标题 |
| 合同开始/结束时间 | 服务期/有效期起止 | "服务期限/合同期限"条款 |
| 合同期限（辅助字段，不占列） | 合同有效期时长，仅当合同写"自签订之日起N年/月"等**相对期限**、无固定结束日时才有值；照抄原文（如"自签订之日起2年"），**不要自己换算成日期**——结束日由代码推算 | "合同期限/有效期"条款 |
| 账期 | 结算/付款账期（天数；非天数表述保留原文） | "结算/付款"条款 |
| 保证金 | 履约保证金/担保金额（单位按目标列） | "履约保证金/履约担保"条款 |
| 税率 | 合同约定增值税率 | "价款/税费/发票"条款 |
| 联系人/电话/地址 | 甲方联系信息 | 末页当事人信息栏/落款 |

**业务条款**（先判"有没有"，有则抽内容；优先看正文专门条款或附则，找不到全文兜底）
- 旺季补偿：是否有 / 时间 / 规则 / 比例
- 油价联动：是否有 / 基准油价
- 疫情补贴：是否有 / 补贴标准
- "是否有X"：合同确认无→`否`；正文未涉及该话题→也可填`否`但标中等置信（与"确认无"区分）。

**邮政明细表**（每条线路一行；**优先位置：附件的价格明细表/中标价格表**，找不到再全文找含逐条线路的表）
- 要抽的列只有：线路名称、邮路性质（单/双程/往返）、里程、分包号、各吨位单价。
- 各吨位单价：**只填表里确实有数字的吨位列**；空格/"/"保持空，缺哪个吨位就 null；**不要替它换算**（换算是代码层的事）。
  其中「4.2米」以车长命名、该车型基本用不到，**识别不确定时直接留空**（标低置信），不要硬凑数字。
- `类型`、`备注`、`合同编号`、`元/趟/条/40吨B` 是红字列 → **不抽取、留空**（见 §3）。即便表里有这些内容也不要填进来。
- 把价格填到**与表头吨位语义对应**的列，不要按列序硬塞；行数 = 表里线路条数；一条都没抽到必须如实报告，不要静默返回空。
- **只抽价格表里直读的线路**。合同常有「比照/延申线路/新增邮路」条款：发往"比照地"城市的线路不在价格表里、由业务按比照公式（协议价÷原里程×新里程）派生，且新里程多来自合同未印出的外部距离表。这类**延申线路不要自行编造**（缺里程会臆造数据）；它由系统单独检测并告警提醒人工补充，不在本次直读范围内。注意别把"比照源/比照地"那种**城市清单表**误当成线路表。

**车型系数表**（**优先位置：附件或正文"车型转换系数/折算"表**，找不到全文兜底）
- 抽出：基准车型、各吨位系数。**只抽表里写明的，不自己计算、不补全缺失吨位。**
- 合同直接逐吨位列价、没有系数表 → 正常情况，返回"未发现系数表"，**不要编造**。

---

## 5. 缺失 / 冲突 / 低置信处置

- 缺失 → null，证据写"未检索到相关表述"。
- 冲突（多处不同值）→ 全列出 + 各自证据 + 低置信。
- 模糊 / OCR 可疑 / 需推断 → confidence ≤ 0.7。
- 始终：**留空或低置信，优于自信地填错。**

## 6. 「邮政」表 29 列逐列口径（权威速查，与 `field_map.yaml` 镜像）

> 处理方式分四类：**直读**=大模型从价格表逐行读；**继承台账**=取自台账同名字段，与台账一致；**派生**=代码按规则算（见 §7）；**不抽取(红字)**=系统外/人工填，留空。（原"固定值=配置默认"已废弃：业务字段一律不用固定默认值。）
> 「红字不抽取」对应模板红字表头与 `field_map.yaml` 的 `extract:false`，**即便表里出现疑似值也一律不填**。

| 列 | 字段 | 处理方式 | 口径要点 |
| --- | --- | --- | --- |
| A | 省 | 继承台账 | 取台账「省」，与台账一致 |
| B | 市 | 派生/继承 | 代码：台账「市」是具体市/区→**继承台账市**（甲方所在地，如台账=南京则全行=南京、台账=大兴则=大兴，不取线路起点）；台账「市」='/'（省级合同）→ 用线路共同起点兜底（全部同起点→该城市，如 33 条「成都-X」→成都；起点跨多市/解析不出→'/'） |
| C | 合同编码 | 不抽取(红字) | 系统外编码，留空 |
| D | 业务类型 | 继承台账 | 取台账「业务类型」（一干/二干/省干/市内…） |
| E | 分包号 | **直读**（红字例外，要抽） | 价格表里的真实数据列，逐行读取 |
| F | 线路名称 | **直读** | 从「主要中标价格明细表/线路价格表/发运计划表」逐条读，**必须输出全部线路、不省略不截断**；线路名命中原文→置信 0.95，未命中→0.5。**别把「比照源城市清单表」当线路表**。行数 = 价格表线路条数（直接决定数据行数） |
| G | 邮路性质 | **直读** | 枚举仅 单程/双程/往返。**枚举外的值（如「单边/双边」）保留原文、标低置信并进复核，不自动归一**；判不出→null |
| H | 里程 | **直读** | 价格表「里程」列数值，不换算 |
| I | 2.75吨/3吨 | 直读+换算 | 主吨位=3，参与系数换算（§7） |
| J | 5吨 | 直读+换算 | 主吨位=5 |
| K | 4.2米 | **直读** | 车长命名、不参与吨位系数换算；**该车型基本用不到，识别不确定时直接留空**（标低置信），不硬凑 |
| L | 8吨 | 直读+换算 | 主吨位=8 |
| M | 12吨/9.6米 | 直读+换算 | 主吨位=12 |
| N | 15吨 | 直读+换算 | 主吨位=15 |
| O | 20吨/12.5 | 直读+换算 | 主吨位=20 |
| P | 25吨 | 直读+换算 | 主吨位=25 |
| Q | 30吨、17.5 | 直读+换算 | 主吨位=30 |
| R | 40吨A | 直读+换算 | 主吨位=40 |
| S | 元/趟/条/40吨B | **不抽取(红字)** | 计价单位/40吨B 列，不直读、不换算，留空 |
| T | 有效期开始时间 | 继承台账 | 取台账「合同开始时间」 |
| U | 有效期结束时间 | 继承台账 | 取台账「合同结束时间」（可能为代码推算值，见 §7） |
| V | 公司主体 | 继承台账 | 取台账「合同主体」（即合同**乙方/承包方**全称，每份合同可能不同：天津智猪网、江西君卓、博华物流…），**不再是固定值** |
| W | 类型 | **不抽取(红字)** | 计趟/计重 × 主/备选，由人工标注，留空 |
| X | 备注 | **不抽取(红字)** | 由人工添加，留空 |
| Y | 是否退线 | 不抽取(红字) | 运营状态，人工维护 |
| Z | 退线时间 | 不抽取(红字) | 运营状态，人工维护 |
| AA | 合同编号 | **不抽取(红字)** | 留空；「同合同不同文件」去重改由文件内容 sha1 保证 |
| AB | 返程线路编码 | 不抽取(红字) | 运营字段，人工维护 |
| AC | 延期结束时间 | 不抽取(红字) | 延期由运营确认后手工补录 |

## 7. 代码层口径（抽取之后由程序执行，大模型不做，仅供核对）

这些不是抽取动作，列在此处是为了让「邮政」表的处理口径集中可查、并与代码（`calculate.py`/`validate.py`）保持一致。

- **吨位系数换算**（`calculate.py`）：有「车型转换系数表」时，基准价 = 某直读吨位价 ÷ 该吨位系数；目标吨位价 = 基准价 × 目标吨位系数（写数值、保留 2 位）；基准吨位系数=1。
  - **自洽校验**：多个直读价反算的基准价互相偏差 > 0.5% → 判 OCR 误读，**放弃换算、只保留直读值**并标复核。
  - 无系数表 → 只保留直读吨位价，其余留空（不补全、不臆造）。
- **市 / 省派生**：省取台账「省」；市按上表 B 列口径——台账市具体则继承台账市，台账市='/'（省级）才用线路共同起点兜底。
- **直辖市「市」归一**（`calculate._normalize_municipality_city`）：直辖市（北京/上海/天津/重庆）合同的台账「市」若被填成'北京市'或留空，据客户名称/地址里紧跟直辖市后的'XX区/县'确定性校正为区县名（'北京市大兴区分公司'→'大兴'）。已是具体区县则不动。
- **公司主体（V列）/合同主体（台账I列）**：取合同**乙方/承包方**全称（抽取得到），明细 V 列继承台账。**严禁**用任何固定默认公司名兜底（历史 bug：硬编码'北京博华物流'，对乙方为'天津智猪网'等合同全错）。
- **有效期结束时间推算**：合同**无固定结束日**但写了相对期限（「自签订之日起 N 年/月」）→ 结束日 = 开始日 + 期限月数 − 1 天；遇「至下一轮采购完成/最多不超过 N 个月」等**延期条款**则截断、只按主期限算，延期部分提示人工确认。
- **校验告警**（`validate.py`，进复核报告顶部）：线路 0 行；价格超区间 / 同列离群（>3×MAD 且偏离>50%）；吨位列混入非数字；**邮路性质枚举外**（如「单边」）；台账省与线路起点市省份不一致；有线路但全无价格（禁止臆造、留空待人工补）。
- **比照 / 延申线路**：合同含「比照/延申/新增邮路」机制时，价格表只直读已列出的线路；发往比照地、里程需按公式由**外部距离表**派生的延申线路，系统**只直读、不臆造**，并给出人工补充提示（缺里程会编造数据）。

---

## 8. 最后提醒

把每份合同当作第一次见到的新模板：只读它写了什么，按字段含义用语义定位，没写的就留空，不要迁移、不要默认、不要脑补。
