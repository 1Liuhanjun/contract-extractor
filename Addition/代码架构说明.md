# 代码架构说明（给协作/合并代码用）

> 面向**接手或合并代码的开发者**。讲清：① 整体数据流；② 每个文件夹/文件负责什么；
> ③ 「邮政明细表」与「新合同台账」两条业务线分别落在哪些文件（合并时最关心）；④ 改规则/扩展时改哪里。
>
> 面向「怎么跑起来」的使用说明见 `README.md`；Web 前后端契约细节见本文件 §7 与 `web/README.md`。

---

## 0. 一句话概览

把扫描版合同 PDF →（托管 PaddleOCR-VL 转成 文字+表格）→（DeepSeek 纯文本大模型抽字段）→
（跨吨位价格换算 / 省市日期派生）→（校验）→（每份合同写一个独立 Excel 台账）→（生成人工复核报告）。

**全程不把图片发给大模型**（无 VLM），靠 OCR 文本 + 规则 + 证据回查保证准确率。

一份合同产出两类数据，对应模板里两张 sheet：
- **新合同台账** sheet：每份合同 **1 行**（合同级字段：甲乙方、期限、税率、保证金、油价/旺季…）。
- **邮政** sheet：每条线路 **1 行**（线路级字段：线路名、里程、各吨位价格…）。

---

## 1. 整体数据流（分层流水线）

`pipeline.process_pdf()`（`pipeline.py`）按第 0~6 层顺序编排，处理一份 PDF：

```
PDF
 └─0 represent  OCR：PaddleOCR-VL → full_markdown + tables(HTML)   [按 sha1 缓存]
 └─1 classify   关键词判公司（邮政/京东/未知）。非邮政→跳过(一期只做邮政)
 └─2 extract    DeepSeek 抽取，三块【并行】：
        ├ extract_ledger        台账字段（两阶段：事实→标准化）   →「新合同台账」sheet 的数据
        ├ extract_routes        线路明细（带价格的线路表）          →「邮政」sheet 的数据
        └ extract_coefficients  车型系数表（用于跨吨位换算）
 └─3 calculate  跨吨位价格换算、省/市派生、合同结束日期推算
 └─(可选) price_ocr  价格数字第二引擎(RapidOCR)交叉复核
 └─4 validate   不变量校验、比照/延申线路提醒，产出告警清单
 └─5 excel_writer 从模板新建一个独立 xlsx，写台账 1 行 + 邮政 N 行
 └─6 report     生成 Markdown 复核报告（留空原因/低置信/候选/换算依据/告警/token 用量）
```

**两个入口**都调同一个 `pipeline.process_pdf()`，只是外围不同：
- **CLI**：`run.py`（命令行批处理 `input/` 目录或指定路径）。
- **Web**：`web/server.py`（FastAPI，上传/实时进度 SSE/历史/下载）。`process_pdf` 多接两个可选参数 `progress`(进度回调) 和 `index_lock`(并发锁)，CLI 不传、行为不变。

## 2. 两条业务线分别落在哪些文件（合并代码重点）

本仓库目前**邮政明细**调优较多、**新合同台账**由同事另行优化。两条线在代码里是**清晰分开**的，合并时按下表对照即可，冲突面很小：

| 关注点 | 🟦 新合同台账线（同事优化区） | 🟩 邮政明细线（本仓库优化区） | 共用 |
| --- | --- | --- | --- |
| LLM 抽取 | `layers/extract.py` → `extract_ledger()`（两阶段） | `layers/extract.py` → `extract_routes()` / `extract_coefficients()` | 同一文件，不同函数 |
| 业务规则源（改规则只改这里） | `companies/ledger.md` + `companies/ledger_kb.py`（读 md 分段拼 prompt） | `companies/youzheng.md`（作为 system prompt 注入） | — |
| 列映射 | `field_map.yaml` 的 `ledger:` 段 | `field_map.yaml` 的 `youzheng:` + `tonnage_columns:` 段 | 同一文件，不同段 |
| 派生计算 | `layers/calculate.py`：省/市派生(`_normalize_municipality_city`/`_derive_city`)、合同结束日期推算 | `layers/calculate.py`：跨吨位换算(`compute` 主体) | 同一文件，不同函数 |
| 写 Excel | `excel_writer.py` → `write_ledger_row()` | `excel_writer.py` → `write_youzheng_rows()` | `write_contract()` 串起两者 |
| 校验 | （台账字段不变量） | `validate.py`：价格区间、比照/延申线路提醒 | `layers/validate.py` |
| 报告 | `report.py` 表一 | `report.py` 表二 | 同一文件 |

> **合并建议**：`extract.py` / `calculate.py` / `excel_writer.py` / `field_map.yaml` 是两条线**共处一文**的地方，合并时按"函数/配置段"粒度合并，而不是整文件覆盖。规则文本（`ledger.md` vs `youzheng.md`）互不相干，可直接各取各的。

## 3. 目录与文件逐个职责

### 顶层入口与配置

| 文件 | 职责 | 注意 |
| --- | --- | --- |
| `run.py` | CLI 入口：收集 PDF → 逐份调 `pipeline.process_pdf` → 每份写独立台账到 `output/contracts/` | 无参时处理 `input/` 下所有 PDF |
| `pipeline.py` | **核心编排**：第 0~6 层顺序调用；三大抽取块（台账/线路/系数）用线程池**并行** | `progress`/`index_lock` 为 Web 而加，CLI 留空 |
| `diag.py` | 联调诊断脚本：分阶段跑、固定日志 `output/diag.log`，首次验证 OCR/LLM 联通用 | — |
| `app_config.py` | 加载 `config.yaml`/`field_map.yaml`/`.env`，解析相对路径，暴露密钥(只从环境变量取) | `Config` 对象贯穿全程 |
| `config.yaml` | OCR/LLM 端点与参数、分类关键词、增强开关、并发度、路径 | `supported_companies: ["邮政"]` 一期只做邮政 |
| `field_map.yaml` | **列映射声明**：台账(`ledger:`)与邮政明细(`youzheng:`)每一列「从哪来/抽不抽/怎么填」；`tonnage_columns:` 吨位列与吨位数 | 改字段/列只改这里，尽量不动代码 |
| `schemas.py` | Pydantic 数据模型：`Field`(值+置信+证据)、`RouteRow`(线路+吨位价格dict)、`Coefficient`(系数表)、`ContractResult`(总结果) | 抽取/计算/写表/报告都围绕这几个结构 |
| `.env` | 两个密钥 `DEEPSEEK_API_KEY` / `PADDLEOCR_ACCESS_TOKEN`，不入库 | — |

### `layers/`　——　流水线各层（每层一个文件，职责单一）

| 文件 | 层 | 职责 | 关键函数 |
| --- | --- | --- | --- |
| `represent.py` | 0 | 调 OCR 得 markdown+表格，按文件 **sha1 缓存并校验**（防同名不同内容串档） | `represent()` / `file_sha1()` |
| `classify.py` | 1 | 关键词命中计数 → 公司(邮政/京东/未知)+置信 | `classify_company()` |
| `extract.py` | 2 | **LLM 抽取（三块）**：见 §2/§4。含截断抢救、证据回查、低置信投票 | `extract_ledger` / `extract_routes` / `extract_coefficients` |
| `verify.py` | 2增强 | 证据回查(evidence 须在原文命中否则降置信)、枚举软归一、self-consistency 投票 | `evidence_hit` / `recheck_evidence` / `self_consistency` |
| `calculate.py` | 3 | 跨吨位换算(基准价×目标系数)、省/市派生、相对期限→合同结束日推算 | `compute` / `_normalize_municipality_city` / `_compute_end_date` |
| `price_ocr.py` | (可选) | RapidOCR 本地二次识别价格表，与托管 OCR 直读价交叉，对不上则降置信进复核 | `cross_check()`（`enhance.cross_check_prices` 开启才跑） |
| `validate.py` | 4 | 不变量校验、价格区间、`比照/延申线路`定价机制提醒（价格表只直读、延申线路人工派生） | `validate` / `check_extension_routes` |
| `excel_writer.py` | 5 | 从模板复制出**独立** xlsx，写台账 1 行 + 邮政 N 行；不确定单元格黄色高亮+批注；价格列补 2 位小数 | `write_contract` / `write_ledger_row` / `write_youzheng_rows` |
| `report.py` | 6 | 生成 Markdown 复核报告（表一台账逐字段、表二线路逐条、告警、token 用量） | `generate()` |

## 4. 第 2 层抽取的内部结构（最核心，重点看）

`pipeline.py` 里这三块**并行**跑（线程池），临界路径 = 最慢一块：

1. **`extract_ledger(llm, skill, field_map, full_md, enhance)` → `dict[字段名→Field]`** 🟦
   两阶段（同事优化重点）：
   - **Stage1 事实抽取**：只问"合同写了什么"，逐条给 value+evidence，不做业务映射。
   - **Stage2 标准化映射**：按 `ledger.md` 的字段知识库把事实映射成标准值（税率→小数、保证金→万元、相对期限→结束日留空…）。
   - 任一阶段异常 → 自动降级回单阶段抽取，绝不"成功地空跑"。
   - 规则全在 `ledger.md`，装配在 `ledger_kb.py`。

2. **`extract_routes(llm, skill, field_map, full_md, tables_md, enhance)` → `(routes, found, None)`** 🟩
   抽「带价格的线路明细表」。**关键判定**（近期修）：合同常同时有"价格明细表"和"发运计划表/比照线路清单"，**存在价格表时只抽价格表、跳过无价清单**；通篇无价格表才回退抽发运计划表。

3. **`extract_coefficients(llm, skill, full_md, tables_md)` → `Coefficient`** 🟩
   抽车型折算系数表（基准车型 + 各吨位系数），供 `calculate.py` 把缺的吨位由直读价反算补全。只抽写明的，不臆造。

> `skill` 形参：三块都由 `pipeline.py` 传入 `cfg.skill_text`(=`youzheng.md`，含反幻觉/反迁移铁律)。台账的**业务规则**来自 `ledger.md`（经 `ledger_kb` 装配进 Stage2），Stage1 把 `skill` 与 `ledger.md` 的事实抽取规则一并注入；线路/系数则直接以 `skill` 作 system prompt。

## 5. 规则与配置：改东西改哪里

| 想改的东西 | 改哪里 |
| --- | --- |
| 台账某字段的取值口径/映射规则 | `companies/ledger.md`（对应分段） |
| 邮政线路/系数的抽取规则 | `companies/youzheng.md` |
| 某一列从哪来/抽不抽/默认值/公式/吨位 | `field_map.yaml`（`ledger:` 或 `youzheng:`/`tonnage_columns:` 段） |
| OCR/LLM 端点、模型、超时、并发、增强开关 | `config.yaml` |
| 跨吨位换算/省市/日期派生逻辑 | `layers/calculate.py` |
| 公司分类关键词 | `config.yaml` 的 `classify:` |

**两条铁律**（已在代码与规则里固化）：
- **不允许业务字段用固定默认值**（历史 bug：合同主体硬编码"北京博华物流"，对乙方是"天津智猪网"的合同全错）。客户分类由分类结果注入，合同主体从乙方抽取。
- **不臆造数据**：合同没印的价格/里程/延期一律留空 + 进复核，不编。

## 6. 几个必须知道的工程约定

- **缓存**：OCR 结果按「文件名 + 内容 sha1」缓存到 `output/_cache/`，命中再校验 sha1 一致才复用 → 重跑同文件秒过 OCR，同名不同内容不会串档。
- **去重**：`_index.json` 按 sha1 记录已处理文件，用于"疑似重复"软告警（不再硬跳过，允许重传重跑）。
- **每份合同独立台账**：`write_contract` 从模板复制一个新 xlsx 只写这一份（不是累加进一个大主表）。CLI 写到 `output/contracts/合同台账_<文件名>.xlsx`，Web 写 `合同台账_<job_id>_<文件名>.xlsx`。
- **硬编码"邮政"**：`classify.py`/`extract.py` 的 prompt 目前按邮政落地；二期加京东需扩展 `pipeline`/`field_map`/新 skill（`config.yaml` 已留 `supported_companies` 与 TODO）。
- **并发**：Web 单 job 内多 PDF 用线程池并发(默认 3，`PIPELINE_CONCURRENCY` 可调)，共享的 `_index.json`/Excel 索引用 `INDEX_LOCK` 串行保护。

## 7. Web 前后端（同事编写区）

**后端 `web/server.py`（FastAPI）**：进程启动初始化一次 `Config/LLM/OCR/INDEX`；每次上传生成 `job_id`，后台线程并发处理该批 PDF；进度用「事件日志 + 条件变量」经 SSE 重放下发（对浏览器重连/StrictMode 双连幂等、不丢事件）。

接口契约：

| 接口 | 方法 | 说明 |
| --- | --- | --- |
| `/api/upload` | POST | 上传 1~N 个 PDF（FormData），落盘后起后台线程，返回 `{job_id, files}` |
| `/api/jobs/{job_id}/stream` | GET | SSE 实时进度，收到 `job_done` 结束 |
| `/api/history` | GET | 读 `_history.json`，返回每一次处理记录（最新在前） |
| `/api/download/excel/{fname}` | GET | 下载某份独立台账（按文件名，防路径穿越） |
| `/api/download/report/{contract_id}` | GET | 下载复核报告（后端拼 `_review.md`） |
| `/` | GET | 生产模式托管前端 `dist/`（开发模式走 vite proxy） |

SSE 事件（前端 `useJobStream.ts` 消费）：`{type: connected/file_start/file_done/job_done}` 或分层进度 `{stage, status, file, ...}`，其中 `stage ∈ {queued,ocr,classify,extract,calculate,validate,write,report,skipped}`、`status ∈ {start,progress,done}`，每条都带 `file` 文件名标签。

**前端 `web/frontend/src/`**（React 18 + Vite 5 + Tailwind 3 + Framer Motion）：
- `App.tsx` 根组件（上传/历史/订阅进度）；`api.ts` 前后端类型契约 + API 客户端；`useJobStream.ts` 把 SSE 事件流 reduce 成 React 状态。
- `components/`：`UploadZone`(拖拽上传)、`PipelineFlow`(七层流水线可视化)、`FileCard`(单文件进度/结果/下载)、`HistoryPanel`(历史表+下载)。

> **前后端契约**集中在 `web/frontend/src/api.ts` 的类型定义 与 `web/server.py` 的路由/事件字段。改了后端事件名/字段要同步改 `api.ts` 的 `Stage`/`StageEvent`/`ProcessResult`。

## 9. 合并代码的建议步骤

1. 先 `git` 各自成支，跑 `pytest -q` 确认两边各自绿。
2. **规则文本**（`ledger.md` ↔ `youzheng.md`）互不相干，直接各取各的，无冲突。
3. **共处一文的代码**按 §2 表格的"函数/配置段"粒度合并：
   - `extract.py`：台账取同事的 `extract_ledger`，邮政取本仓库的 `extract_routes`/`extract_coefficients`。
   - `field_map.yaml`：`ledger:` 段取同事的，`youzheng:`/`tonnage_columns:` 段取本仓库的。
   - `calculate.py`/`excel_writer.py`/`report.py`：台账相关函数 vs 邮政相关函数分别取。
4. **前端 + `web/server.py`** 取同事版本（整体由其编写）；注意若 `pipeline.process_pdf` 的事件名/参数有改动，需同步 `api.ts`。
5. 合并后再跑 `pytest -q` + 用 `diag.py` 真机过一份样例，确认台账与邮政两张表都对。
