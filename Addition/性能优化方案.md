# 合同读取智能体 · 性能优化方案（保准确率 + 单份 <60s）

> 版本：v1 ｜ 日期：2026-06-04 ｜ 适用代码：`方案2 ocr 合并/代码`
> 目标：在**不牺牲交付物（含线路价格明细表）和准确率**的前提下，把单份合同的"提交→出结果"压到 **60 秒以内**。

---

## 0. TL;DR（先看这段）

- **瓶颈 100% 在 LLM，不在 OCR。** 实测：武汉(44页) OCR 仅 59s，但 6 次 LLM 调用花了 ~321s；安徽(26页) OCR 32s，LLM ~233s。
- **第一性原因：你开着 DeepSeek 思考模式，每次调用 40–53s；同事显式关掉，每次 5–8s。** 单这一项就是 6–8 倍差距，且对"抄字段"类抽取几乎不损准确率。
- **第二性原因：你做的事是同事的 3 倍。** 同事只填"新合同台账"主表（2 次调用），**完全不做**线路价格明细表、车型系数表、自洽投票。所以"30s vs 3min"不是同题对比。
- **结论：不要砍交付物去凑 30s。** 正确路径＝先抄"关思考"这个送分项，再把多出来的活**并行化 + 瘦身**，即可在保住线路抽取的前提下进 60s。
- 本方案分 **P0/P1/P2** 三级落地，每级都有**独立验证 + 回滚**，并配一套**多重验证机制**（计时埋点 + 黄金集回归 + 单旋钮 A/B + 准确率红线）。

## 1. 背景与实测基线

### 1.1 真实耗时拆分（来自 `output/logs/run.log`）

| 合同 | 页数 | OCR 耗时 | LLM（次数 / 耗时 / token） | 总时长 |
|------|------|---------|---------------------------|--------|
| 武汉 | 44 | **59s** | 6 次 / **~321s** / 257,396 | 380s |
| 安徽安庆 | 26 | **32s** | 6 次 / **~233s** / 145,838 | 265s |
| 青岛-贵阳（网页上传，两阶段路径） | 42 | — | **21 次** / — / 364,362 | 307s |

> 关键观察：
> 1. 每次 LLM 调用平均 **39–53 秒**，这只有"思考模式 + 巨大输入 + 大 max_tokens"才会出现。
> 2. 单份 token 量 14–36 万，因为**整份 OCR 文本（≤120K 字符）被重复发送 4+ 次**（事实抽取、线路、系数、每轮投票各发一遍）。
> 3. 21 次那份是两阶段路径（事实+标准化）叠加线路表超长触发**截断重试/抢救**，调用数被放大。

### 1.2 同题对比 vs 不同题对比

| 维度 | 同事（contract-extractor） | 你（方案2 ocr 合并） |
|------|---------------------------|---------------------|
| 思考模式 | **关**（`thinking:{type:disabled}`） | **开**（默认） |
| 主流程 LLM 调用数 | **2**（Stage1 事实 + Stage2 标准化） | 6（单阶段路径）～21（两阶段+截断重试） |
| 线路价格明细表 | **不做** | 做（`extract_routes`） |
| 车型系数表 | **不做** | 做（`extract_coefficients`） |
| 自洽投票 | **不做** | 做（`n_votes=3`，+3 次/份） |
| 价格二次 OCR 交叉 | 不做 | 可选（当前 `cross_check_prices:false`，已关） |
| `max_tokens` | 4K–8K | 32K（全局统一） |
| 派生字段（合同名称/编码/账期/预警） | **纯 Python** 算 | 部分靠 LLM |
| 单份耗时 | ~30–50s | ~180–380s |

**批判性结论：** 同事快，一半是"关思考"这个纯工程优化（你能直接抄），另一半是"他只解主表这道小题"（你不能照抄删除，否则丢线路价格交付物）。本方案据此把优化拆成"送分项"和"架构项"两类分别处理。

## 2. 根因分析（按贡献度排序）

| 序号 | 根因 | 证据位置 | 量级影响 |
|------|------|---------|---------|
| R1 | **思考模式默认开启**，每次调用先生成长思维链 | `config.yaml:23,30`；`providers/llm.py` 未发送 `thinking` 字段 | 单次 ~45s → ~7s，**最大头** |
| R2 | **6–21 次串行调用**，三大独立块（台账/线路/系数）+ 投票全部一个等一个 | `pipeline.py:91–102`、`layers/verify.py:116` | 串行 = 各块耗时之和 |
| R3 | **全量文本重复发送 + max_tokens=32K** | `extract.py:_clip`、`llm.py:33` | 输入/输出 token 翻几倍，单次更慢、易截断 |
| R4 | **投票固定 3 次**，且在慢模型上重采样 | `config.yaml:58–61`、`verify.py:100` | +3 次/份 |
| R5 | **截断→抢救→重试**放大调用数（大线路表） | `llm.py:_salvage_truncated`、`max_retries=3` | 个别合同调用数 ×3 |

## 3. 优化措施（P0 / P1 / P2 分级）

> 原则：**一次只动一个旋钮，改完即测**（见 §5 多重验证）。每条都给"改动点 / 预期 / 风险 / 验证 / 回滚"。

### P0 —— 送分项（低风险、立竿见影，先做）

#### P0-1 关闭思考模式 ✅ 最高优先级
- **改动点**：`providers/llm.py` `DeepSeekProvider.call_json`，`payload` 增加：
  ```python
  payload["thinking"] = {"type": "disabled"}   # 与同事一致
  ```
  并把 `config.yaml` 的 `llm.timeout` 由 300 回收到 120。
- **预期**：单次调用 45s→~7s；整份 LLM 从 ~270s→~45s。**仅此一步大概率进 60s。**
- **风险**：少数"需推理"的派生字段（合同类型由中标人判定、账期=结算+交付）准确率可能下降——因为思考模式此前可能在**替弱 prompt 兜底**。
- **验证**：§5 黄金集跑一遍，重点盯 `合同类型 / 账期 / 省市 / 旺季`，与开思考的基线逐字段对比。
- **回滚**：删掉该行即恢复。
- **配套**：把这些派生字段的判定规则**写进 prompt/知识库**（参考同事 `field_knowledge_base` + `main.py` 里的 Python 规则），让"关思考"后仍稳——这是 P0-1 能否不掉准确率的关键。

#### P0-2 分块设置 max_tokens
- **改动点**：`call_json(..., max_tokens=None)` 增加形参，`_safe_call` 透传；各块按需传：台账/系数 **6K**，线路 **16K**（线路多时仍够，避免截断抢救）。
- **预期**：减少无谓生成时间，消除 R5 的截断重试放大。
- **风险**：线路表极多时 16K 仍可能不够 → 仍有 `_salvage_truncated` 兜底；必要时该合同单独提到 24K。
- **验证**：检查日志中"输出达到 max_tokens 被截断"告警是否归零；线路条数与原表核对。
- **回滚**：恢复统一 32K。

#### P0-3 确认价格二次 OCR 保持关闭
- **现状**：`cross_check_prices:false`，本就未跑。**保持关**（开启会在本地再跑一遍 RapidOCR 渲染，明显加时）。仅在离线抽检时手动开。

> **P0 验收门槛**：典型合同（10–30 页）单份 ≤60s，且黄金集字段准确率相对基线**不下降**（红线见 §5.4）。达标才进 P1。

---

### P1 —— 架构项（中风险、收益大，P0 达标后做）

#### P1-1 三大块并行
- **改动点**：`pipeline.py` 把 `extract_ledger` / `extract_routes` / `extract_coefficients` 放进线程池并发（三者互不依赖）。
- **临界路径**：由"三块之和"变为"最慢的一块"。台账链（Stage1→Stage2→投票）通常是临界路径；线路块输入最大，是单次最慢调用。
- **预期**：LLM 临界路径 ~45s → ~20s。
- **风险**：线程安全。`DeepSeekProvider` 的 `call_count/total_tokens += ` 非原子（仅统计，误差可忽略，可加锁消除）；`index` 已有 `index_lock`。OCR/LLM 均为 IO 密集，`ThreadPoolExecutor` 适用。
- **验证**：并发版与串行版在黄金集上**输出逐字段一致**（并行不应改变结果，只改变墙钟）。
- **回滚**：恢复顺序调用。

#### P1-2 投票并行
- **改动点**：`verify.self_consistency` 的 `for i in range(n_votes)` 改为并发提交后汇总。
- **预期**：投票从 3×7s=21s → ~7s，**不损准确率**（票数与逻辑不变）。
- **风险**：低。
- **验证**：同一合同并行/串行投票的多数票结果一致。
- **回滚**：恢复循环。

#### P1-3 线路/系数块输入瘦身
- **改动点**：线路块优先只喂 `tables_md`（OCR 表格区），仅当表格为空才回退全文；系数块同理。减少每次重发整份 `full_md`。
- **预期**：线路/系数单次调用提速，token 下降。
- **风险**：极少数合同线路写在正文非表格里 → 保留"表格为空才喂全文"的回退，避免漏抽。
- **验证**：黄金集线路条数/系数与基线一致。
- **回滚**：恢复喂 `full_md`。

> **P1 验收门槛**：典型合同 LLM 临界路径 ≤25s，端到端（含 OCR）典型 ≤55s；准确率不低于 P0。

### P2 —— 调优项（按需，数据说话再做）

| 措施 | 触发条件 | 说明 |
|------|---------|------|
| P2-1 投票降级 | P0/P1 后准确率达标且仍想再快 | `n_votes` 3→1，或仅对"证据未命中"字段投票（同事不投票都达标，说明可调） |
| P2-2 线路+系数合并为 1 次调用 | 想再省 1 次调用 | 二者都读价格/系数表；合并需控输出长度，仅小表合同适用 |
| P2-3 利用 DeepSeek 上下文缓存 | 想降 input 延迟/成本 | 保持 system（skill）稳定且置于前部，投票重发命中 `prompt_cache_hit`，观察日志缓存命中率 |
| P2-4 OCR 页范围裁剪 | 仅超大合同(>40页)且 60s 是硬上限 | 借鉴同事 `PADDLEOCR_PAGE_RANGES`，**有漏页风险，需人工确认页范围**，默认不启用 |

## 4. 目标时间预算（改造后）

```
单份典型合同（10–30 页）：
┌─ OCR（托管异步，页数相关，基本固定） ──────────── 25–40s
└─ LLM（P0+P1 后，并行临界路径）
     台账链  Stage1(7s) → Stage2(7s) → 投票并行(7s) = 21s   ← 临界路径
     线路块  ~15–20s  ┐
     系数块  ~5s      ┘ 并行隐藏在台账链之下
   LLM 合计 ≈ 21s
端到端 ≈ 46–61s  → 典型合同达标 ✅
```

**诚实的边界声明（必须让相关方知晓）：**
- **超大合同（≥40 页）受 OCR 物理下限制约**：武汉 44 页光 OCR 就 59s，单这一项已逼近 60s。对这类合同，硬性 <60s 只能靠 P2-4 页裁剪（有漏页风险）或**接受其为例外**。建议把 SLA 定义为"典型合同 <60s，超大合同尽力而为并提示页数"。
- OCR 与 LLM 在**单份内无法重叠**（LLM 依赖 OCR 输出）；批量上传时已有并发，提升的是吞吐而非单份延迟。

## 5. 多重验证方案（本文件的核心，务必执行）

> 优化最大的坑是"快了但偷偷变不准"。下面四层验证缺一不可。

### 5.1 计时埋点（让每一步耗时可证）
- **现状**：`pipeline.py` 只记录总时长 + 总 token，无法区分 OCR vs 各 LLM 块。
- **改动**：在每个 `emit(stage=...)` 边界记录 `time.time()`，输出到复核报告，得到 `OCR / 台账 / 线路 / 系数 / 投票 / 写表` 各段墙钟。
- **价值**：每次旋钮调整的提速来源可归因，避免"以为是 A 起效其实是 B"。

### 5.2 黄金回归集（准确率红线）
- **构成**：选 **5–8 份**已有人工参考答案/复核报告的合同，覆盖：省级/市级、有/无系数表、线路多/少、有/无旺季油价。建议含：青岛-贵阳、安徽安庆、武汉、浙江 + 2 份省级。
- **基线**：在**未改任何代码前**，用现版本（开思考）跑一遍，保存每份的字段输出与线路价格，作为 `baseline.json`。
- **比对器**：复用同事 `verify_results` 思路（数值等价、日期格式等价、"/"与空等价、长文本模糊匹配）做逐字段 diff，输出准确率 %。

### 5.3 单旋钮 A/B（一次只动一个变量）
按 P0-1 → P0-2 → P1-1 → P1-2 → P1-3 顺序，**每改一项**：
1. 跑黄金集；
2. 记录"端到端时长、LLM 临界路径、字段准确率、线路命中率、token"五个指标；
3. 与上一步对比，确认"时间下降且准确率不降"才进下一项；否则定位该旋钮的副作用再决定。

### 5.4 验收红线（Go/No-Go）
| 指标 | 红线 | 不达标动作 |
|------|------|-----------|
| 主表字段准确率 | **≥ 改造前基线**（不允许净下降） | 回滚该旋钮 / 补 prompt 规则 |
| 线路价格逐条命中率 | ≥ 基线 | 提高线路块 max_tokens / 喂表格优先 |
| 系数表正确率 | = 基线 | 回滚瘦身 |
| 典型合同端到端 | ≤ 60s | 进 P1/P2 |
| 关键字段（价格、税率、账期、保证金、合同时间） | **零退化** | 一票否决，必须修复 |

> 特别提醒：**价格/金额类字段是一票否决项**。本系统无 VLM 兜底，价格精度只能靠 OCR + 抽取层，关思考后务必重点抽检价格逐条。

## 6. 分阶段实施计划

| 阶段 | 内容 | 产出 | 门槛 |
|------|------|------|------|
| 阶段0 | 建黄金集 + 跑开思考基线 + 加计时埋点 | `baseline.json`、分段计时 | 基线可复现 |
| 阶段1（P0） | 关思考 + 分块 max_tokens + 补派生字段规则 | 单旋钮 A/B 报告 | 典型≤60s 且准确率不降 |

每阶段结束**留存一次黄金集报告**，形成可追溯的优化档案。

## 7. 风险登记表

| 风险 | 等级 | 缓解 |
|------|------|------|
| 关思考后派生字段（合同类型/账期）变差 | 中 | 把判定规则写进 prompt/知识库；黄金集盯防；可单字段回退思考 |
| 线路表过大被截断丢行 | 中 | 线路块 max_tokens=16K + 保留 `_salvage_truncated`；超大表单独提额 |
| 并行引入隐蔽数据竞争 | 低 | LLM 统计加锁；并行/串行输出一致性测试 |
| 超大合同 OCR 即超 60s | 中 | 重新定义 SLA（典型 vs 超大）；P2-4 页裁剪需人工确认 |
| 价格字段静默退化 | 高 | 列为一票否决；每次 A/B 必抽价格逐条 |
| 缓存/瘦身改动漏喂正文导致漏抽 | 低 | "表格空才回退全文"双保险 |
