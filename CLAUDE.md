## 一、项目概述

### 核心目标
将物流运输外包合同的 **OCR扫描件** → 自动提取 **21个关键字段** → 填入固定 Excel 模板。

### 业务背景
- 发包方：中国邮政各分公司（甲方）
- 承包方（乙方，我方）：北京博华物流有限公司 或 天津智猪网网络科技有限公司（不同合同不同）
- 合同类型：一级干线邮件运输外包服务协议
- 原工作：人工阅读 40+ 页合同 → 手动填 30+ 个 Excel 字段 → **全自动化是目标**

## 二、技术架构

### 整体流程

```
PDF扫描件 → PaddleOCR → .docx/.txt 文本
    ↓
┌─────────────────────────────────────────────────────────────┐
│                 两阶段提取系统 (Web 应用)                    │
│                                                             │
│  Stage 1:  LLM 事实抽取                                     │
│  输入：合同OCR文本                                          │
│  输出：{字段名: {value: "内容", evidence: "原文引用"}}        │
│  核心问题："合同里写了什么？"  不做业务推断                    │
│                                                             │
│  Stage 2:  LLM 标准化映射                                    │
│  输入：Stage 1 事实 + 字段知识库 + 少样本实例                  │
│  输出：21个标准 Excel 字段值                                  │
│  核心问题："业务人员看到了这个事实，会把它填成什么？"            │
└─────────────────────────────────────────────────────────────┘
    ↓
累积 Excel (_results/合同提取汇总.xlsx) — 每次提取自动追加行
```

### 关键技术决策

| 决策 | 原因 |
|------|------|
| **两阶段分离** | Stage 1 只做阅读理解，Stage 2 只做规则映射。两者各司其职，错误可追溯 |
| **字段知识库集中管理** | 所有映射规则、允许值、少样本实例都在 `field_knowledge_base.py`，业务变化时只改这一个文件 |
| **少样本实例独立存储** | `few_shot_examples.py` 存已核对的 "事实→标准值" 对，与程序逻辑解耦 |
| **urllib 替代 httpx** | httpx 在遇到中文时触发 ASCII 编码错误，换用标准库解决 |
| **DeepSeek V4 Pro 禁用思考模式** | V4 Pro 默认开启 thinking，导致 content 为空；加 `thinking: {type: disabled}` |

## 四、关键规则约定

### 4.1 字段分类 (2026-06-02 业务确认版 v2)

| 类别 | 数量 | 说明 |
|------|:---:|------|
| 🟢 **LLM 提取字段** | 20 个 | Stage 1→2 从合同原文提取 + 规则映射 |
| 🔵 **代码自动生成** | 4 个 | 登记日期、合同编码、合同名称、合同预警提醒（Python 算） |
| 🔴 **不填字段** | 8 个 | 项目编号、是否完成签订、是否同步财务、联系人、电话、地址、快递单号、钉钉审批单号 |

完整规则见 `字段提取业务规则手册.md`。

### 4.2 核心业务规则 (2026-06-02 确认 v2)

**账期** = 结算周期 + 交付周期。两段原文分别提取，两个数相加。一般 15+30=45。
**合同时间**：中间"合同期限/有效期"段落优先 > 末尾签章。两处都有只看中间。
**合同类型（⚠️ 双维度 + 乙方名校验）**：
  维度1—乙方的中标人描述（只提取包含乙方公司名的段落，甲方描述全部丢弃）
  维度2—线路表格"线路性质"列（正式/备用）
  最终：主选侧✅+备选侧✅→主选/备选，只有主选→主选，只有备选→备选
**合同编码**：`BHWL-YZ邮政-YYYYMMDDNNN`
**合同名称**：第一条线路名 + "一干运输合同"
**合同预警提醒**：结束时间 - 今天 = 剩余天数（负数=已过期）
**合同主体**：从合同"承包方（乙方）："提取（不硬编码，可能是博华或天津智猪网等）
**省/市 直辖市规则**：北京/上海/天津/重庆 → 省填城市名，市填区名（如省`北京`、市`大兴`）
**保证金**：元÷10000 转万元。无具体金额留 `/`
**旺季**：条款有且生效（打勾/未划）→填；未打勾→不填（后续时间/规则/比例也不填）
**油价基准**：只填数字不管单位（元/升 or 元/吨）
**疫情补贴**：不只找"补贴"字面，疫情导致的价格调整（如"临时启用旺季价"）也算"是"
**补贴标准**：有明确数字/公式才填，模糊表述（"单独申请"）→ `/`
**空值格式**：空缺字段输出 `/`（匹配人工填写习惯）

### 4.3 Python 级安全校验（LLM 多次误判后的加固）

| 校验项 | 位置 | 逻辑 |
|------|------|------|
| **中标描述主语校验** | `process_contract` | 中标/成交人描述不含乙方公司名 → 标记 `_filtered=True`，Stage 2 不可用 |
| **合同类型回退** | `process_contract` | 如中标信息全被过滤，只看线路表维度判断合同类型 |
| **复核前过滤** | `_handle_llm_review` | 传给复核 LLM 前，中标描述做同样过滤，标注 `[已过滤]` |

### 4.3 验证规则 (语义宽松匹配)

| 场景 | 例子 | 判定 |
|------|------|:---:|
| `/` 与空字符串 | 补贴标准="/" vs AI="" | ✅ 一致 |
| `否` 与 `/` | 疫情补贴="否" vs 参考="/" | ✅ 一致 |
| `9%` 与 `0.09` | 税率 | ✅ 数值等价 |
| `10%` 与 `10.00%` | 补偿比例 | ✅ 数值等价 |
| `"合肥邮区中心"` vs `"合肥邮区中心局"` | 客户名称 | ✅ 模糊匹配 |
| `"30天"` vs `"45天"` | 账期 | ❌ 不一致 |
| 标红字段 | 联系人/登记日期等 | ⛔ 不参与验证 |

### 4.3 已验证的映射规则

| 字段 | 规则 |
|------|------|
| **客户分类** | 甲方名称含"邮政"→"邮政"、含"京东"→"京东" |
| **业务类型** | 合同标题含"一级干线"→"一干" |
| **合同类型** | 框架协议+中标项目→"主选" |
| **税率** | 一般纳税人+运输服务→9% |
| **保证金** | 合同写"XXX元" → ÷10000 转万元 |
| **合同开始时间** | 取最后一个签署/盖章日期 |
| **合同结束时间** | 开始时间+有效期年数-1天 |

|------|------|

| 客户名称"局"字——合同写"邮区中心"vs"邮区中心局" | 🟡 待确认 |

## 六、LLM 复核机制

### 流程
```
提取完成 → 上传参考答案Excel → 验证Tab选对应行
    → 报告面板 → LLM复核Tab → 🚀 执行复核
    → 自动保存复核档案到 _reviews/
```

### 复核输出
- **裁定结果**：AI正确 / 参考答案正确 / 都不正确
- **原文位置**：合同第几条第几款
- **原文摘录**：合同原句逐字引用
- **复核依据**：法官式说理
- **存档文件**：`时间戳_省份城市_复核报告.md` + `.json`

## 八、已知限制 & 注意事项

1. **前端状态不持久化**：刷新浏览器会丢失验证数据和参考答案，需重新上传
2. **逐份处理模式**：上传新合同时清空旧的，累积 Excel 行自动追加到 `_results/合同提取汇总.xlsx`
3. **答案 Excel**：必须有 `新合同台账-2026年5月` Sheet（系统读最后一个）
4. **不填字段**：登记日期/项目编号/是否完成签订/是否同步财务/联系人/电话/地址/快递单号/钉钉审批单号共 9 个不提取不验证
5. **少样本不自动更新**：必须我明确要求才更新 `few_shot_examples.py`
6. **合同类型有 Python 硬校验**：中标人描述不含乙方名则被过滤，只看线路表维度
